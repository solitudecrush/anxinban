# 智慧养老项目后端互通概述

## 一、后端整体定位

后端是系统的**业务与数据中枢**，基于 **Spring Boot + Spring Data JPA** 构建，对外提供统一的 REST API（响应格式 `ApiResponse`，HTTP 状态码始终为 200）。负责：

- 管理老人档案、家属用户、工作人员账号
- 接收并持久化传感器健康数据（心率、体温、血氧、血压、睡眠等）
- 处理告警事件（跌倒、心率异常、闯入、门锁异常等）
- 下发设备控制指令（灯光、门锁、音乐播放等）
- 管理视频监控权限、服务申请、工单流转、通知
- 与本地智能体（边缘网关）和云端智能体进行数据同步与指令交互

------

## 二、后端 ↔ 前端（Web 管理端 + 家属 App）

前端分为 **Web 管理端**（社区工作人员）和 **家属 App**，后端通过 REST API 提供服务。

### 2.1 Web 管理端

| 功能模块     | 典型接口                                                     | 说明与状态                                                   |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 认证         | `POST /api/auth/login`                                       | 工作人员登录，返回 token（未做强制校验， 生产环境需补充 JWT） |
| 老人管理     | `GET /api/elder/list`                                        | 分页查询老人列表，支持按姓名、楼栋、健康状态筛选             |
| 老人详情     | `GET /api/elder/{elderId}`                                   | 返回完整档案（基本信息、最新健康、设备列表、最近告警、最近工单） |
| 健康数据     | `GET /api/health/latest/{elderId}`                           | 最新体温、心率、血压、血氧、睡眠等级与入睡时间               |
| 健康趋势     | `GET /api/health/trend/{elderId}`                            | 按 day/week/month 获取指定指标的历史曲线                     |
| 告警管理     | `GET /api/alarm/list` `PUT /api/alarm/{id}/resolve`          | 告警列表查询、确认处理；闯入告警单独接口 `/api/alarm/intrusion/list` |
| 工单管理     | `GET /api/work-order/list` `POST /api/work-order` `PUT /api/work-order/{id}/assign` | 工单创建、分页查询、分配处理人、状态更新                     |
| 服务申请     | `GET /api/service-request/list` `POST /api/service-request/{id}/convert` | 查看家属提交的申请（上门看护/设备维修等），可转为工单或拒绝  |
| 监控申请     | `POST /api/monitor-request` `GET /api/monitor-request/check` | 工作人员向家属发起监控查看申请，家属在 App 端审批            |
| 社区看板     | `GET /api/dashboard/stats`                                   | 老人总数、今日告警数、在线设备数、待处理工单数等统计         |
| 设备管理     | `GET /api/device/list`                                       | 设备列表，含在线状态、电量、安装位置                         |
| 工作人员管理 | `GET /api/staff/list` `POST /api/staff`                      | 工作人员账号 CRUD，默认密码 123456                           |
| 文件上传     | `POST /api/upload/avatar` `POST /api/upload/snapshot`        | 头像、告警快照图片上传（当前仅返回模拟 URL）                 |

### 2.2 家属 App

| 功能模块   | 典型接口                                                     | 说明                                               |
| :--------- | :----------------------------------------------------------- | :------------------------------------------------- |
| 认证       | `POST /api/auth/login`                                       | 家属登录，绑定老人 ID                              |
| 绑定老人   | `GET /api/elder/bound?familyId=`                             | 获取家属关联的老人信息                             |
| 健康数据   | `GET /api/health/latest/{elderId}` `GET /api/health/trend/{elderId}` | 实时健康 + 趋势图表                                |
| 健康档案   | `GET /api/health-record/by-elder/{elderId}`                  | 查看住院史、病史、过敏史、常用药物等               |
| 服务申请   | `POST /api/service-request` `GET /api/service-request/my-list` | 提交上门看护、设备维修、健康咨询等申请，并查看历史 |
| 监控审批   | `GET /api/monitor-request/list/family` `POST /api/monitor-request/{id}/approve` | 查看工作人员的监控申请，同意/拒绝/撤销权限         |
| 紧急联系人 | `GET /api/emergency-contact/list` `POST /api/emergency-contact` | 管理老人的紧急联系人                               |
| SOS 求救   | `POST /api/sos` `GET /api/sos/list`                          | 触发 SOS 并查看历史记录                            |
| 通知       | `GET /api/notification/list` `GET /api/notification/unread-count` | 获取闯入告警、申请状态等通知，前端需轮询实现实时性 |

### 2.3 前端交互要点

- **统一响应**：所有接口返回 `{ code, message, data }`，外层 HTTP 状态码始终为 200。
- **分页**：列表接口使用 `page`、`pageSize`，返回 `PageResult{ list, total, page, pageSize }`。
- **时间格式**：ISO8601 字符串（`yyyy-MM-ddTHH:mm:ss`）。
- **权限控制**：当前无强制 Token 校验， 生产环境必须补充 Spring Security + JWT。

## 三、后端 ↔ 硬件（ESP32 设备 + 本地智能体）

硬件包括：**手表端 ESP32**（健康监测、一键呼叫）、**房屋端 ESP32**（环境传感器、门锁、蜂鸣器、压力传感器）、**三路摄像头**（客厅、卧室、大门）。

### 3.1 设备注册与状态

| 接口                                | 说明                                                         |
| :---------------------------------- | :----------------------------------------------------------- |
| `POST /api/device/register`         | 设备首次上线，上报 `deviceId`、`elderId`、`deviceType`。返回设备对象。 |
| `PUT /api/device/{deviceId}/status` | 定时上报 `online`/`offline` 及 `lastHeartbeat`，后端更新设备在线状态。 |

### 3.2 传感器数据上报

- **直连后端**：`POST /api/device/{deviceId}/sensor-data`，支持批量 `sensorDataList`。
- **通过本地智能体批量转发**：`POST /api/local-agent/data`（携带 `agentId`、`deviceId`、`sensorDataList`），后端调用同一逻辑写入数据库。
- 传感器类型包括 `heart_rate`、`temperature`、`blood_oxygen`、`insomnia`、`sleep_time` 等。 **血压数据**（`blood_pressure_sys`/`blood_pressure_dia`）仅存入 `sensor_data` 表，不会自动汇总到 `blood_pressure` 表，需后续补充写入逻辑或由云端另行写入。

### 3.3 告警上报

- 硬件或本地智能体检测到异常（跌倒、心率过高、烟雾、门锁异常等）：
  - 推荐调用 `POST /api/local-agent/alarm-report`（支持自动生成 `alarmId`，默认 `severity=medium`）。
  - 也可直接调用 `POST /api/alarm`。
- 后端将告警持久化至 `alarm_event` 表，并可通过通知接口被前端获取。

### 3.4 设备控制指令

1. **下发命令**：前端或云端调用 `POST /api/device/{deviceId}/command`，命令存入内存中的 pending 队列（`commandStore`），返回命令对象。
2. **拉取命令**：本地智能体轮询 `GET /api/local-agent/{agentId}/commands`。 **当前实现返回所有设备的 pending 命令**。本地网关需自行根据 `deviceId` 筛选属于自己的命令。
3. **上报结果**：执行后调用 `POST /api/local-agent/control` 记录执行日志。

### 3.5 摄像头与视频流

- 权限检查：`GET /api/monitor-request/check?elderId=&staffId=`，返回是否有监控权限。
- 获取推流地址：`GET /api/elder/{elderId}/camera-stream`。**当前返回硬编码 RTSP 地址**，未对接真实流媒体服务器（如 ZLM、SRS）。生产环境需替换为动态生成的鉴权地址。

### 3.6 本地智能体专用接口

| 接口                                      | 用途                       |
| :---------------------------------------- | :------------------------- |
| `POST /api/local-agent/heartbeat`         | 维持网关在线状态，自动注册 |
| `POST /api/local-agent/data`              | 批量转发传感器数据         |
| `POST /api/local-agent/alarm-report`      | 上报告警                   |
| `GET /api/local-agent/{agentId}/commands` | 拉取待执行命令             |
| `POST /api/local-agent/intent`            | 上报语音意图识别结果       |
| `POST /api/local-agent/control`           | 上报家居控制执行日志       |

## 四、后端 ↔ 云端（云端智能体 + 云服务）

云端包括：**云端智能体**（负责长期健康分析、AI 对话、音乐疗法推荐）、**第三方云平台**（告警同步、配置下发）。

### 4.1 云端智能体注册与状态

| 接口                                    | 说明                                 |
| :-------------------------------------- | :----------------------------------- |
| `POST /api/cloud-agent/register`        | 云端智能体启动时注册，提供 `agentId` |
| `POST /api/cloud-agent/report`          | 定时上报心跳、设备数、故障信息       |
| `GET /api/cloud-agent/{agentId}/config` | 拉取配置（心跳间隔、告警阈值等）     |

### 4.2 数据与告警同步

- 云端智能体可将外部告警同步至系统：`POST /api/cloud-agent/alarm`（与硬件告警共用告警库）。
- 获取老人综合上下文（用于 AI 决策）：`GET /api/agent/context?elderId=`，返回最近健康、最近3条告警、干预、对话、意图。

### 4.3 干预任务

| 接口                                                  | 说明                           |
| :---------------------------------------------------- | :----------------------------- |
| `POST /api/intervention`                              | 创建干预任务（如主动音乐疗法） |
| `GET /api/cloud-agent/{agentId}/interventions`        | 拉取待执行干预（最多100条）    |
| `POST /api/cloud-agent/{agentId}/intervention-result` | 上报干预执行结果               |

### 4.4 AI 对话与建议

- 保存对话记录：`POST /api/cloud-agent/chat`。 `agentReply` 目前基于规则生成（`ModelGatewayService.ruleBasedResponse`），未接入真实 LLM。
- 生成 AI 健康建议：`POST /api/cloud-agent/advice`。

### 4.5 云平台联动

- 第三方云平台可通过调用 `/api/cloud-agent/alarm` 将告警推入本系统，再由本系统通知前端。
- 反向推送（本系统告警主动通知第三方云）暂未实现，需根据业务需求补充。

## 五、核心数据流

### 5.1 健康数据流

```
硬件传感器 → 本地智能体 → POST /api/local-agent/data
                           ↓
                   后端写入 sensor_data 表
                           ↓
           前端通过 GET /api/health/latest 展示
```

### 5.2 告警数据流

```
硬件/边缘网关检测异常 → POST /api/local-agent/alarm-report
                              ↓
                    后端生成 alarm_event 记录
                              ↓
        前端轮询 GET /api/notification/unread-count 获取新告警
                              ↓
              工作人员调用 PUT /api/alarm/{id}/resolve 处理
```

### 5.3 控制指令流

```
前端/云端 → POST /api/device/{id}/command
                    ↓
          后端存入内存 pending 队列
                    ↓
  本地智能体轮询 GET /api/local-agent/{id}/commands（ 全局返回）
                    ↓
          本地网关执行硬件动作 → POST /api/local-agent/control 上报结果
```

### 5.4 监控授权流

```
工作人员 → POST /api/monitor-request
                ↓
     家属 App → POST /api/monitor-request/{id}/approve
                ↓
   后端更新 elder_user.camera_auth_until 和 monitor_request.status
                ↓
  工作人员调用 GET /api/elder/{id}/camera-stream（ 返回硬编码地址）
```

## 六、关键约定与注意事项

| 项目         | 约定                                                         |
| :----------- | :----------------------------------------------------------- |
| 统一响应     | `{ "code": 200, "message": "success", "data": {...} }`，HTTP 状态码始终为 200 |
| 业务 ID 类型 | 所有 ID 均为 String（如 `elder_001`, `alarm_123`）           |
| 时间格式     | `yyyy-MM-ddTHH:mm:ss`（示例：`2025-12-19T10:00:00`）         |
| 分页参数     | 默认 `page=1, pageSize=20`，返回 `PageResult{list, total, page, pageSize}` |
| 血压数据     | 硬件上报 `blood_pressure_sys`/`blood_pressure_dia` 仅存 `sensor_data`，最新血压需单独写入 `blood_pressure` 表 |
| 主动音乐疗法 | 设计上由压力传感器检测上床后触发，当前 `InterventionController` 仅支持手动创建干预，需后续实现自动触发策略 |
| 鉴权         | 当前未强制 Token，生产环境必须增加 JWT 或设备签名            |
| 命令持久化   | 当前存储在内存，重启丢失，建议改为数据库表                   |
| 视频流地址   | 硬编码示例，需对接真实流媒体服务器                           |
